# First Principles: Minimum Viable Definition

## What's the Absolute Minimum?

**Core Principle**: Before building anything, ruthlessly identify what's truly essential. Most failures come from overbuilding, not underbuilding.

## MVP vs. MLP Definition

### MVP: Minimum Viable Product

> The simplest version that can be released to test core hypotheses

**Characteristics**:
- Tests ONE primary assumption
- Contains ONLY what's needed to test
- Accepts imperfection in non-critical areas
- Designed to be learned from and iterated

**MVP Test**:
```
Can this version test our core hypothesis?
Yes/No: _____________________________

What happens if this fails?
→ _____________________________

Can we build this faster/better with less?
→ _____________________________
```

### MLP: Minimum Lovable Product

> The simplest version that creates genuine value and delight

**Characteristics**:
- Users actually want to use it
- Creates positive emotional response
- Quality where users feel it
- May be slightly more than "minimum" in specific areas

**MLP Test**:
```
Would a real user love this?
Yes/No/Unsure: _____________________________

What would make them love it?
→ _____________________________

What would they forgive as "rough edges"?
→ _____________________________
```

## The User Story Minimum

### Thinking from User Outcomes

**Instead of**: What features should we build?

**Ask**: What user outcomes do we enable?

### User Story Framework

```
As a [type of user],
I want to [achieve specific outcome],
So that [measurable benefit].

Minimum viable version of this story:
→ _________________________________
```

**Example**:

| User Story | Full Version | Minimum Viable Version |
|------------|--------------|----------------------|
| As a busy professional, I want to track my tasks, so I never forget anything | Full project management with calendar sync, recurring tasks, collaboration, reporting | Simple list with checkboxes |

## Validation Checklist

### Before Finalizing the Minimum

- [ ] **Core question test**: Can this version answer our core question?
- [ ] **Pain threshold test**: Will early adopters tolerate this?
- [ ] **Feature audit**: Have we removed everything non-essential?
- [ ] **Cost-benefit test**: Is each element worth its cost?
- [ ] **Time test**: Is this achievable in the time available?
- [ ] **Iteration test**: Can we learn and iterate from this?
- [ ] **Failure test**: If this fails, do we know why?
- [ ] **Success test**: If this succeeds, do we know what to do next?

## The Anti-Pattern Warning

### Signs You're NOT at Minimum

- [ ] You're excited to build more than what's essential
- [ ] You can't explain why each feature is in the MVP
- [ ] You think users will expect feature X (not test need)
- [ ] You're designing for the "final" version
- [ ] You can't build and launch in your time constraint
- [ ] You're optimizing for impressing stakeholders

### Signs You Might Be TOO Minimum

- [ ] You can't explain the value proposition
- [ ] Users don't understand what to do
- [ ] You can't measure anything meaningful
- [ ] The product doesn't work end-to-end
- [ ] Early users are turned away, not just tolerant

---

*Remember: The goal is to build the smallest thing that teaches you the most. Less is more—but only if the "less" still teaches you what you need to know.*
