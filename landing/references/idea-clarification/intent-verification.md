# User Intent Verification Checklist

## Purpose

The User Intent Verification Checklist is designed to systematically uncover, validate, and document the true intentions behind user requests. This document emphasizes the critical role of verification in preventing misunderstandings and ensuring project success.

**Why Intent Verification is Critical:**
- Surface hidden requirements before development begins
- Align stakeholders on shared understanding
- Prevent expensive pivots mid-project
- Build foundation for accurate scope definition
- Ensure delivered solutions match user expectations

## Part 1: Understanding Confirmation Questions

### 1.1 Basic Comprehension Verification

#### Direct Understanding Check
Use these questions to confirm basic comprehension:

**Question Pattern A - Paraphrasing:**
"Can I summarize what I heard as: [restate in your own words]?"

**Question Pattern B - Specific Confirmation:**
"When you mention [key term], are you referring to [specific interpretation]?"

**Question Pattern C - Scope Check:**
"Does this apply to [specific context] or is it broader than that?"

### 1.2 Context Understanding

**Questions to Clarify Context:**

1. **Background Understanding:**
   ```
   - What is the current workflow/process you follow?
   - How does this request fit into your daily operations?
   - What systems or tools are currently in use?
   - Who else is involved in this process?
   ```

2. **Trigger Understanding:**
   ```
   - What prompted this request now?
   - Has this been an ongoing issue or a new development?
   - What changed that made this necessary?
   ```

3. **Environment Understanding:**
   ```
   - Are there any technical constraints we should be aware of?
   - Are there regulatory or compliance considerations?
   - What is the user environment (desktop/mobile/specific devices)?
   ```

### 1.3 Priority Understanding

**Questions to Clarify Priorities:**

1. "If we could only achieve one thing from this request, what would it be?"

2. "How does this rank compared to your other ongoing projects?"

3. "Is there a hard deadline we must meet?"

4. "What aspects can be compromised if necessary, and what cannot?"

### 1.4 Stakeholder Alignment Verification

**For multi-stakeholder requests:**

| Stakeholder | Role | Primary Need | Verification Status |
|-------------|------|--------------|---------------------|
| | | | ☐ Aligned ☐ Misaligned |
| | | | ☐ Aligned ☐ Misaligned |
| | | | ☐ Aligned ☐ Misaligned |

**Verification Questions:**
- "Have all stakeholders agreed on this requirement?"
- "Are there any dissenting opinions we should address?"
- "Who has final approval on this deliverable?"

## Part 2: Expectation Clarification Questions

### 2.1 Output Expectation

**Questions to Define Expected Output:**

1. **Format Expectations:**
   ```
   - What format would you like the final deliverable in?
   - Should this be a document, system, process, or something else?
   - Are there specific templates or examples we should follow?
   ```

2. **Quality Expectations:**
   ```
   - What level of polish/quality is expected?
   - Is this a prototype, MVP, or production-ready solution?
   - Are there specific quality standards to meet?
   ```

3. **Completeness Expectations:**
   ```
   - What does "done" look like for this project?
   - Are there any specific deliverables you expect?
   - Should this be a standalone solution or integrate with existing systems?
   ```

### 2.2 Success Expectation

**Questions to Define Success:**

1. "How will you measure if this solution is successful?"

2. "What specific metrics or KPIs should improve?"

3. "What problem will no longer exist once this is complete?"

4. "How will you know if the solution exceeded expectations?"

### 2.3 Timeline Expectation

**Questions to Clarify Timeline:**

1. "What is the expected duration for this project?"

2. "Are there intermediate milestones or checkpoints?"

3. "Is there flexibility in the timeline if needed?"

4. "What dependencies exist that could affect the timeline?"

### 2.4 Resource Expectation

**Questions to Clarify Resources:**

1. "What resources (budget, personnel, tools) are available?"

2. "Are there any constraints on who can be involved?"

3. "What support can we expect from your team?"

4. "Are there external vendors or partners involved?"

### 3.2 Techniques for Mining Implicit Intent

#### Technique 1: The "What's Really Going On?" Approach
```
Questions to Ask:
- "What will this enable you to do that you can't do now?"
- "What does solving this problem mean for your larger goals?"
- "What have you tried before, and why didn't it work?"
- "What would happen if you didn't solve this problem?"
```

#### Technique 2: The Negative Framing Approach
```
Questions to Ask:
- "What would an unacceptable solution look like?"
- "What should we definitely avoid doing?"
- "What are you hoping we don't do?"
- "What would disappoint you about this project?"
```

#### Technique 3: The Future Vision Approach
```
Questions to Ask:
- "Imagine we succeed beyond expectations. What does that look like?"
- "If this problem is solved perfectly, how does your work change?"
- "What does your ideal end state look like?"
- "What becomes possible after this is complete?"
```

### 3.3 Common Implicit Intents to Surface

| Implicit Intent | Questions to Uncover |
|----------------|---------------------|
| **Cost reduction** | "What budget constraints exist?" / "What costs are you trying to reduce?" |
| **Time savings** | "Where does time currently go to waste?" / "How much time should this save?" |
| **Risk mitigation** | "What risks are you concerned about?" / "What could go wrong?" |
| **Career advancement** | "How does this project affect your goals?" / "What success here means for you?" |
| **Stakeholder impressing** | "Who are you reporting this to?" / "What will make this impressive?" |
| **Process control** | "What aspects do you need to control?" / "What autonomy is needed?" |
| **Job security** | "How does this affect your role?" / "What skills are being utilized?" |

### 3.4 Assumption Surfacing

**Common Assumptions to Verify:**

| Assumption | Verification Question |
|------------|----------------------|
| User knows what they want | "Have you done something like this before?" |
| Solution approach is correct | "Have you considered alternative approaches?" |
| Stakeholders are aligned | "Have all affected parties been consulted?" |
| Timeline is realistic | "What happens if we need more time?" |
| Budget is adequate | "What if this costs more than estimated?" |
| Technical feasibility | "Are there any technical limitations we should know about?" |

## Part 4: Comprehensive Verification Framework

### 4.1 Verification Session Structure

**Opening (5 minutes):**
```
1. Set the stage: "My goal is to make sure I understand your needs completely."
2. Confirm agenda: "I have [X] key areas I want to explore."
3. Establish ground rules: "There are no wrong answers here."
```

**Body (30-45 minutes):**
```
1. Start with broad questions
2. Move to specific details
3. Probe for implicit needs
4. Validate understanding throughout
```

**Closing (10 minutes):**
```
1. Summarize key findings
2. Confirm next steps
3. Schedule follow-up if needed
4. Document decisions made
```

Before proceeding to solution design, verify:


**If any checkbox is unchecked, do not proceed until resolved.**

## Part 5: Red Flags and Warning Signs

### 5.1 Indicators of Misunderstanding

- User seems hesitant when you paraphrase their request
- Stakeholders give conflicting information
- Requirements keep changing during discussion
- User uses vague language ("improve", "optimize", "enhance")
- User focuses on solutions rather than problems
- There is pressure to skip the clarification phase

### 5.2 Indicators of Hidden Agendas

- User seems to have predetermined solution
- Certain topics are avoided or glossed over
- Some stakeholders are absent from discussions
- User has strong emotional reactions to specific topics
- Requests seem politically motivated
- User is vague about who will use the solution

### 5.3 Recovery Strategies

**When misalignment is detected:**
1. Stop and acknowledge the gap
2. Ask clarifying questions
3. Use different question approaches
4. Bring in additional stakeholders
5. Document the misunderstanding
6. Recalibrate expectations
