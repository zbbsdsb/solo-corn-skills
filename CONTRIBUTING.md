# Contributing to SOLO CORN SKILLS

**Thank you for considering contributing to SOLO CORN SKILLS!** Every contribution, no matter how small, helps this project grow.

## Code of Conduct

This project adheres to a code of conduct. By participating, you are expected to uphold this code.

### Our Pledge
We pledge to make participation in our community a harassment-free experience for everyone, regardless of age, body size, disability, ethnicity, sex characteristics, gender identity and expression, level of experience, education, socio-economic status, nationality, personal appearance, race, religion, or sexual identity and orientation.

### Our Standards
Examples of behavior that contributes to creating a positive environment include:
- Using welcoming and inclusive language
- Being respectful of differing viewpoints and experiences
- Gracefully accepting constructive criticism
- Focusing on what is best for the community
- Showing empathy towards other community members

## How Can I Contribute?

### Reporting Bugs
Before creating bug reports, please check the [existing issues](../../issues) to avoid duplicates. When you create a bug report, include as many details as possible.

**A good bug report should include:**
- A clear and descriptive title
- Steps to reproduce the issue
- Expected behavior
- Actual behavior
- Screenshots (if applicable)
- Environment details (OS, browser, etc.)

### Suggesting Enhancements
Enhancement suggestions are tracked as GitHub issues. Provide the following information:
- A clear and descriptive title
- A detailed description of the proposed enhancement
- Explanation of why this enhancement would be useful
- Examples of how this enhancement would be used

### Code Contributions

#### Prerequisites
- Git
- Basic understanding of Git workflows

#### Your First Code Contribution
If you're new to open source, start with "good first issue" labeled issues.

#### Pull Request Process
1. Fork the repository (see [FORK.md](FORK.md) for detailed instructions)
2. Create a feature branch: `git checkout -b feature/your-feature-name`
3. Make your changes following our [coding standards](#coding-standards)
4. Commit your changes: `git commit -m 'Add some feature'` (see [commit message guidelines](#commit-message-guidelines))
5. Push to the branch: `git push origin feature/your-feature-name`
6. Open a Pull Request

## Coding Standards

### General Guidelines
- Keep it simple and focused
- Follow the Brutalist Linear Identity design philosophy
- Write clear, readable code
- Comment only when necessary (explain why, not what)

### Documentation Standards
- Use Markdown for all documentation
- Follow the existing documentation structure
- Keep documentation concise and actionable

## Commit Message Guidelines

Follow the Conventional Commits specification:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation only changes
- `style`: Formatting, white-space, etc.
- `refactor`: Neither fixes a bug nor adds a feature
- `test`: Adding or correcting tests
- `chore`: Build process or auxiliary tool changes

**Example:**
```
feat(mental-models): add new decision-making model

Add a new cognitive framework for improved decision analysis.
Includes reference documentation and integration notes.

Closes #123
```

## Pull Request Guidelines

- PRs should be focused (one feature/fix per PR)
- Include a clear description of the change
- Link to any related issues
- Ensure the PR follows our coding standards
- Keep PRs small and manageable

## Review Process

1. Maintainers will review your PR as soon as possible
2. Address any feedback provided
3. Once approved, your PR will be merged
4. You will be added as a contributor

## Getting Help

If you need help, feel free to:
- Open an issue with your question
- Reach out to the maintainers
