# Skill Combination Recommendations

This document provides guidance on when, how, and why to combine multiple skills from the Solo skill ecosystem to achieve optimal outcomes.

## When to Combine Skills

### Decision Matrix

| Condition | Recommendation |
|-----------|---------------|
| Problem spans multiple domains | Combine relevant skills |
| Need both analysis and implementation | Combine `research-integration` + `product-builder` |
| Product requires market validation | Combine `research-integration` + `marketing-growth` |
| Building and launching full product | Combine all three skills |
| Problem is contained in single domain | Use single skill |

### Combination Triggers

**Strong Triggers (Always Combine):**
- Full product development and launch
- Business planning from concept to market
- Strategic decisions with implementation needs
- Market entry with product准备好了

**Moderate Triggers (Consider Combining):**
- Complex decision with multiple frameworks needed
- Product iteration based on market feedback
- Growth strategy with technical requirements
- Academic research applied to practical implementation

**Weak Triggers (Single Skill Usually Sufficient):**
- Simple, contained problems
- Time-critical single-focus tasks
- Clear, well-defined requirements
- Independent problem domains

## Skill Combination Patterns

### Pattern 1: Research → Build → Market

**Use Case:** Complete product lifecycle from idea to customer

```
┌─────────────────────────────────────────────────────────────┐
│                    PRODUCT LAUNCH FLOW                       │
└─────────────────────────────────────────────────────────────┘

Phase 1: Research & Validation
├── Skill: research-integration
├── Duration: Initial phase
└── Output: Validated concept, strategic frameworks

         ↓

Phase 2: Product Development  
├── Skill: product-builder
├── Duration: Core development
└── Output: Working product, deployment ready

         ↓

Phase 3: Market Launch
├── Skill: marketing-growth
├── Duration: Launch and iteration
└── Output: Customer acquisition, growth
```

**Example:**
```
User: "I want to build a SaaS tool for task management and get my first 100 customers"

→ Phase 1: research-integration
   - Lean Startup methodology
   - Market analysis for productivity tools
   - Competitive positioning
   
→ Phase 2: product-builder
   - Product specification
   - Frontend/backend development
   - Testing and deployment
   
→ Phase 3: marketing-growth
   - Go-to-market strategy
   - Content marketing plan
   - Customer acquisition campaigns
```

## Execution Order Guidelines

### Sequential Execution

**Rule 1: Analysis Before Action**
Always use `research-integration` before implementation skills when:
- Decisions have significant consequences
- Multiple options need evaluation
- Strategic alignment is important

**Rule 2: Build Before Market**
Always complete product development before marketing when:
- Product doesn't exist yet
- Features need to be finalized
- Customer-facing elements depend on product

**Rule 3: Validate Before Scale**
Always research before large marketing investments when:
- Channel effectiveness is unknown
- Market sizing is uncertain
- Customer acquisition costs are unclear

### Sequential Order Matrix

| Problem Type | Recommended Order | Rationale |
|--------------|-------------------|-----------|
| Full product launch | RI → PB → MG | Research validates, build creates, market launches |
| Marketing strategy | RI → MG | Research informs strategy |
| Technical decision | RI → PB | Research guides architecture |
| Feature planning | MG → PB | Marketing insights shape features |
| Market expansion | MG → RI | Marketing learnings guide research |
| Product iteration | MG → RI → PB | Feedback informs direction |

*RI = research-integration, PB = product-builder, MG = marketing-growth*

### Transition Points

**Research → Product Builder Transition:**
- Validated concept document exists
- Key decisions have been made
- Technical requirements are clear
- Strategic framework is selected

**Product Builder → Marketing Growth Transition:**
- Product is functional and deployed
- Core features are finalized
- Target customer is identified
- Value proposition is defined

**Marketing → Research Transition:**
- Unexpected market feedback received
- Growth plateau requires strategic pivot
- New competitive threats identified
- Customer insights require deeper analysis

## Anti-Patterns to Avoid

### Over-Combination

**Anti-Pattern 1: Analysis Paralysis**
```
Problem: Using RI excessively before PB or MG
Symptoms: Endless research, no implementation
Resolution: Set research phase deadlines
```

**Anti-Pattern 2: Premature Marketing**
```
Problem: Marketing before product is ready
Symptoms: Poor conversion, reputation damage
Resolution: Define minimum viable product first
```

**Anti-Pattern 3: Parallel Overload**
```
Problem: Too many skills running simultaneously
Symptoms: Conflicting outputs, integration challenges
Resolution: Limit parallel execution to 2 skills max
```

### Under-Combination

**Anti-Pattern 4: Single Skill Limitation**
```
Problem: Using only one skill for complex problems
Symptoms: Gaps in solution coverage
Resolution: Assess problem complexity upfront
```

### Misalignment

**Anti-Pattern 5: Handoff Gaps**
```
Problem: Skills not properly aligned at transitions
Symptoms: Rework, inconsistencies
Resolution: Define clear outputs for each phase
```

## Decision Framework for Combinations

### Quick Decision Tree

```
START: Problem Analysis
│
├─ Is this a complete business problem?
│   ├─ YES → Use all three skills (RI → PB → MG)
│   └─ NO → Continue
│
├─ Does this require strategic decisions?
│   ├─ YES → Add research-integration first
│   └─ NO → Continue
│
├─ Does this require implementation?
│   ├─ YES → Add product-builder
│   └─ NO → Continue
│
└─ Does this require customer acquisition?
    ├─ YES → Add marketing-growth
    └─ NO → Use single skill
```
